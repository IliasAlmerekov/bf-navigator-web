# Project Convention (Hackathon MVP Mode)

## 1. Goal

This convention is optimized for: - fast team onboarding - rapid MVP
delivery - clean and readable code - avoiding chaos during hackathon

Not optimized for: - long-term enterprise scaling - heavy abstractions

---

## 2. Tech Stack (fixed)

- React + Vite
- TypeScript (strict mode)
- TanStack Router (file-based routing)
- Zustand (client state)
- TanStack Query (server state)
- Axios (API)
- Vitest + React Testing Library
- ESLint + Prettier

---

## 3. Core Principles

### 3.1 Simplicity \> Abstraction

- Do not overengineer
- Avoid premature abstraction
- Write code for current use case

### 3.2 Single Responsibility

- Components → UI only
- Hooks → connect UI + data
- Services → API + basic logic
- Store → client state only

### 3.3 Predictability

- Same structure everywhere
- Same naming everywhere

---

## 4. Project Structure

    src/
      routes/        # TanStack Router file-based route definitions
      pages/         # Page components (UI logic)
      components/
        ui/
        layout/
      hooks/
      services/
      store/
      types/
      constants/
      utils/
      routeTree.gen.ts  # Auto-generated by TanStack Router — do not edit

---

## 5. Page Structure

Each page component lives in `src/pages/`, each route definition in `src/routes/`.

    src/pages/FeaturePage/
      FeaturePage.tsx
      FeaturePage.module.css
      index.ts
      FeaturePage.test.tsx (optional)

    src/routes/feature-page.tsx   ← thin file, imports from pages/

### Rules

- Default = 3--4 files
- Additional files allowed if needed
- Do NOT force structure if it slows you down

---

## 6. Naming Rules

### Files

- PascalCase → components, pages
- camelCase → hooks, utils, services, store

---

## 7. Components

- Pure UI (no API calls)
- No business logic
- No direct service usage

---

## 8. Hooks

- Bridge UI ↔ data
- Can use services, React Query, Zustand

---

## 9. Services

- Handle API calls
- Basic transformation
- Return typed data

---

## 10. State Management

- Zustand → UI state
- React Query → server state
- Do NOT mix

---

## 11. Types

    types/
      api.ts
      domain.ts

---

## 12. Styling

- Use CSS Modules (.module.css) or Tailwind
- Use design tokens / CSS variables
- Avoid inline styles

---

## 13. Accessibility (MANDATORY)

- aria-label required where label is not visible
- aria-role required on interactive elements
- No color-only meaning
- Click targets ≥ 44px
- Support browser text scaling

---

## 14. Error Handling

- No raw API errors in UI
- Show user-friendly messages

---

## 15. Testing (MVP)

- Test only critical flows
- Component tests optional

---

## 16. MVP Scope Rules

Build order: 1. Happy path 2. Loading 3. Error 4. Polish

---

## 17. Imports Order

1. React
2. Libraries
3. Internal
4. Types
5. Styles

---

## 18. Code Quality

- No any
- No unused code
- Small functions
- Clear naming

---

## 19. Git Convention

feat: add feature\
fix: bug fix\
refactor: code improvement

---

## 20. Definition of Done

- Works end-to-end
- No crashes
- Accessible
- Typed
- Readable

---

## 21. Post-Hackathon

- Move to feature-based structure
- Separate domain logic
- Add server-side rendering if needed
